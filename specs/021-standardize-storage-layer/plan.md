# Implementation Plan: Standardise Storage Layer

**Branch**: `021-standardize-storage-layer` | **Date**: 2026-04-17 | **Spec**: [spec.md](./spec.md)  
**Research**: [research.md](./research.md) | **Data Model**: [data-model.md](./data-model.md) | **Contracts**: [contracts/storage-contracts.md](./contracts/storage-contracts.md) | **Quickstart**: [quickstart.md](./quickstart.md)

## Summary

Introduce a typed class hierarchy (`BaseStorage → IDBStorage → entity classes`) that replaces the current boilerplate-heavy plain-object adapters. Create 9 entity TypeScript interfaces in `core/types/entities/`. Build `core/storage/` as a new infrastructure layer with IDB implementations. For Electron, replace `electron-store` with `better-sqlite3` via an abstract `SQLite3Storage` base and 9 SQLite-backed persist collections. Keep Query Builder state as renderer-local localStorage UI state, and treat `persist:merge-all` as a required Electron restore contract for backup import.

## Technical Context

**Language/Version**: TypeScript 5.x  
**Framework**: Nuxt 3.16 (SPA, `ssr: false`), Vue 3.5, Electron 30+  
**Primary Dependencies**:
  - Web storage: `localforage ^1.10.0` (already installed)  
  - Electron storage: `better-sqlite3` (needs `bun add -D better-sqlite3 @types/better-sqlite3`)  
  - Native rebuild: `electron-rebuild` (needs config in `postinstall` + `electron-builder.yml`)  
  - No Kysely — raw better-sqlite3 is sufficient; avoids extra dependency  
**Testing**: Vitest 4 (unit, `node` environment), `fake-indexeddb` for IDB tests  
**Target Platform**: Web browser (Chrome) + macOS/Windows/Linux (Electron 30+)  
**Project Type**: Hybrid web + desktop application (database editor)  
**Constraints**:
  - Storage classes MUST have zero Vue/Nuxt imports (framework-independent)  
  - SQLite code stays in `electron/` — never imported by `core/` (browser safety boundary)  
  - All IPC channel names unchanged (backward compat with existing Electron renderer code)  
  - `TabView` MUST stay `type` alias (not `interface`) — Atlassian d&d library constraint  
**Scale/Scope**: ~9 entity types, ~11 SQLite tables, ~9 IDB storage classes, ~9 SQLite storage classes
